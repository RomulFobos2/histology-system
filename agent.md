# agent.md

## Контекст

- Проект: АИС ГИ, автоматизированная система гистологических исследований для судебно-медицинской экспертизы.
- Назначение: учёт дел и образцов, работа с микроскопическими изображениями, заключениями, протоколами, отчётами и электронным журналом.
- Стек: Java 21, Spring Boot, Spring Security 6, Spring Data JPA/Hibernate, Thymeleaf, MapStruct, Lombok, MySQL, Apache POI, Maven, Python-микросервис автоэнкодера.
- `application.properties` задаётся локально и не должен считаться источником проектных правил.

## Структура проекта

- Основной Java-код: `src/main/java/ru/mai/histology/`.
- Плоские пакеты без подпакетов для `dto/`, `enumeration/`, `mapper/`, `models/`, `repo/`.
- Контроллеры разделены по ролям: `controllers/employee/admin`, `head`, `histologist`, `laborant`, плюс `general`.
- Сервисы разделены по ролям и общим задачам: `service/employee/...` и `service/general`.
- Шаблоны: `src/main/resources/templates/blocks`, `general`, `employee/...`.
- Статика только локальная: `src/main/resources/static/css`, `js`, при необходимости `vendor`.
- Python-часть автоэнкодера живёт отдельно в `autoencoder/`.

## Роли и доступ

- `ROLE_EMPLOYEE_ADMIN`: управление пользователями.
- `ROLE_EMPLOYEE_HEAD`: отчёты, надзор, судебно-медицинские заключения.
- `ROLE_EMPLOYEE_HISTOLOGIST`: образцы, изображения, заключения гистолога, протоколы, автоэнкодер.
- `ROLE_EMPLOYEE_LABORANT`: дела, образцы, загрузка изображений, электронный журнал.
- Доступ строится по URL-префиксам `/employee/admin/`, `/employee/head/`, `/employee/histologist/`, `/employee/laborant/`.

## Какие файлы читать перед работой

2. При любом написании кода читать `CODESTYLE.md`.
3. `TEMPLATES.md` читать только если создаётся новая сущность с нуля.
4. `PLAN.md` использовать для этапов, статуса и понимания общей дорожной карты.

## Правила разработки

- Архитектура: `Controller -> Service -> Repository -> Entity`, DTO и Mapper идут отдельным слоем.
- Controller должен быть тонким: принимает параметры, вызывает сервис, готовит `Model`, возвращает шаблон или `redirect:`.
- Вся бизнес-логика находится в service.
- Repository работает только с данными.
- DTO должны быть обычными классами с `@Data`, не `record`.
- Mapper через MapStruct с `Mappers.getMapper(...)`, без Spring `componentModel`.
- Для новых сущностей соблюдать порядок: Entity -> Repository -> DTO -> Mapper -> Enum -> Service -> Controller -> HTML -> navigation -> seed.

## Жёсткие ограничения

- Не использовать `@Autowired`, только конструкторную инъекцию.
- Не бросать исключения из service: возвращать `Optional.empty()` или `false`.
- Каждый `@Transactional` метод оформлять через `try-catch` и при ошибке вызывать `TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()`.
- Не использовать `@RestController` для HTML-страниц.
- Не использовать jQuery, только vanilla JS.
- Не подключать CDN или любые внешние CSS/JS URL.
- Не создавать подпакеты внутри `dto`, `mapper`, `models`, `repo`.
- Не использовать `th:object` и `th:field`.
- Не менять существующую цветовую схему CSS без явной причины и отдельного запроса.

## Соглашения по коду и шаблонам

- Таблицы БД именовать с префиксом `t_` и camelCase.
- Entity: `@Data`, `@NoArgsConstructor`, `@Entity`, `@Table`, `@EqualsAndHashCode(of = "id")`.
- На коллекциях и обратных ссылках в entity обязателен `@ToString.Exclude`.
- Для полей entity нужен Javadoc на русском.
- `@PathVariable(value = "id") long id` писать именно в таком виде.
- Имена `@RequestParam` начинать с `input`.
- После успешного POST делать redirect на details-страницу, при ошибке возвращать форму, при отсутствии записи уводить на список.
- Для уникальных полей обязательна AJAX-проверка с ответом `{"exists": boolean}`.
- Перед удалением нужен `confirmDelete()`.
- HTML-страницы строить вокруг `header -> container.my-5 -> profile-card -> footer`.
- CSS/JS подключать только из локальных файлов.

## Бизнес-правила

- Пользователей не удалять, а деактивировать через `isActive = false`.
- Нельзя деактивировать собственную учётную запись и учётные записи с ролью администратора.
- Отчество не обязательно: в БД хранить пустую строку `""`, а не `null`.
- Сброс пароля администратором должен выставлять `needChangePassword = true`.
- После входа пользователи попадают на `/`, без редиректа по роли.
- Стадии образца идут строго последовательно: `RECEIVED -> FIXATION -> EMBEDDING -> SECTIONING -> STAINING -> MICROSCOPY -> ANALYSIS -> COMPLETED`.
- При переходе на `MICROSCOPY` статус образца становится `AWAITING_ANALYSIS`.
- При создании заключения гистологом жизненный цикл образца идёт через `ANALYZED` к `CONCLUDED`.
- Каждая роль должна видеть и изменять только свой разрешённый функционал.
- Автоэнкодер работает как отдельный Python REST-сервис, а результат улучшения сохраняется как новое изображение со ссылкой на оригинал.

## Этапы проекта

- Этап 0: фундамент, security, начальные данные, UI-блоки.
- Этап 1: управление пользователями и отделами.
- Этап 2: судебные дела и образцы.
- Этап 3: управление изображениями.
- Этап 4: заключения гистолога и протоколы.
- Этап 5: судебно-медицинские заключения.
- Этап 6: автоэнкодер и интеграция.
- Этап 7: отчёты и аналитический дашборд.
- Этап 8: электронный журнал и финальная полировка.
- По текущему `PLAN.md` все этапы 0-8 пока помечены как не начатые.
