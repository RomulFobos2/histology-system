# Разработка Python-микросервиса для улучшения изображений

> Документ объясняет, **что именно было спроектировано и написано вручную** при интеграции нейросети, а что взято из библиотек «как есть». Это нужно, чтобы на защите честно и конкретно ответить на вопрос «какую работу вы сделали сами?».

---

## 1. Зачем вообще нужен отдельный Python-сервис

Вся основная система — Spring Boot на Java 21. Но нейросети и тяжёлые операции с изображениями живут в экосистеме Python: PyTorch, Pillow, NumPy. Портировать это в Java нельзя — в Java нет сопоставимых по зрелости инструментов для свёрточных сетей.

Поэтому принято архитектурное решение:

- Нейросетевая часть вынесена в **отдельный микросервис** (папка `autoencoder/`).
- Сервис поднимается на FastAPI + Uvicorn.
- Java общается с ним по HTTP через `RestTemplate`.
- Изоляция: тяжёлую ML-часть можно перезапустить, не останавливая основную систему.

Всё это — кирпичи. Само по себе ни одна из этих библиотек не знает, как улучшать гистологические снимки. Задача была собрать из кирпичей работающую систему — именно это наша инженерная работа.

## 4. Что мы написали самостоятельно

### 4.1 Архитектура U-Net (`model/autoencoder.py`)

Из кирпичей `torch.nn` мы собрали конкретную архитектуру под задачу «взять плохой гистологический снимок и вернуть чистый».

**Три наших класса-блока:**

1. **`DoubleConv`** — две последовательные свёртки 3×3 с BatchNorm и ReLU между ними. Базовая единица обработки признаков. Мы сами решили, что именно **двойная** свёртка, именно с ядром 3, именно с паддингом 1, именно с BatchNorm.

2. **`DownBlock`** — `DoubleConv` + MaxPool2d(2). Уменьшает разрешение вдвое и увеличивает число каналов. Особенность: блок возвращает **два тензора** — до пулинга (для skip connection) и после (для передачи дальше). Эту логику написали мы.

3. **`UpBlock`** — `ConvTranspose2d` для увеличения разрешения вдвое + конкатенация с соответствующим skip-тензором + `DoubleConv`. В нашей реализации есть обработка случая, когда размеры после транспонированной свёртки не совпадают с skip-тензором (padding/crop) — для этого добавлен fallback на `F.interpolate`.

**Класс `HistologyDenoisingUNet`** — это и есть наша модель. Топология:

```
Вход: [B, 3, H, W]  (RGB)
  │
  ├─ Down1: 3  → 32  каналов,  разрешение ÷2   ┐
  ├─ Down2: 32 → 64  каналов,  разрешение ÷2   │ skip connections
  ├─ Down3: 64 → 128 каналов,  разрешение ÷2   │
  │                                             │
  ├─ Bottleneck: 128 → 256 каналов (без пулинга)│
  │                                             │
  ├─ Up3: 256+128 → 128 каналов,  разрешение ×2 ┘
  ├─ Up2: 128+64  → 64  каналов,  разрешение ×2
  ├─ Up1: 64+32   → 32  каналов,  разрешение ×2
  │
  └─ FinalConv 1×1: 32 → 3 канала
```

Всё это **мы скомпоновали сами**. Количество блоков (3 вниз + bottleneck + 3 вверх), количество каналов на каждом уровне (32/64/128/256), решение использовать skip connections, место bottleneck — всё это наши проектные решения.

**Финальный residual (ключевой момент):**

```python
def forward(self, x):
    ... # encoder, bottleneck, decoder
    y = torch.sigmoid(self.final_conv(d1))
    return torch.clamp(x * 0.35 + y * 0.65, 0.0, 1.0)
```

Это не «чистый» U-Net — мы сделали так, что сеть не рисует картинку с нуля, а **предсказывает поправку к входу**. Коэффициенты 0.35 / 0.65 подобраны опытно:

- Если взять только `y` (0/1), сеть иногда теряет исходные структуры клеток.
- Если взять только `x`, сеть не учится.
- 0.35·in + 0.65·out — компромисс: поправка доминирует, но исходный сигнал не пропадает.

Это наш инженерный выбор, в стандартных статьях такого микса нет.

### 4.2 Пайплайн синтетической деградации (`HistologyImageDataset._degrade_image`)

**Проблема:** у нас есть только «хорошие» снимки (чистые кадры микроскопии). А обучать нужно на парах `(плохой, хороший)` — иначе сеть не узнает, что такое «плохой».

**Наше решение:** из каждого хорошего снимка генерируем «плохой» искусственно, прогоняя его через цепочку случайных искажений. Сеть учится возвращать оригинал — и в результате умеет чистить реальные снимки.

Цепочка из 6 шагов (каждый применяется с вероятностью, указанной в скобках):

| Шаг | Что делаем | Параметры | Вероятность |
|---|---|---|---|
| Downscale | Уменьшение и обратное увеличение | масштаб 0.45–0.85× | 90% |
| Gaussian blur | Расфокус | радиус 0.4–1.8 | 85% |
| Contrast | Сжатие/растяжение контраста | множитель 0.75–1.15 | 80% |
| Brightness | Сдвиг яркости | множитель 0.85–1.1 | 70% |
| Гауссовский шум | Случайные значения на пиксели | σ = 4–18 | 90% |
| JPEG-артефакты | Сжатие и повторное открытие | quality factor 20–60 | 60% |

Все диапазоны и вероятности — **наши**. Функция `_degrade_image` полностью написана вручную — она использует только базовые операции Pillow и NumPy.

### 4.3 Функция потерь (гибридная)

Стандартная L1-loss сравнивает пиксели. Но для гистологии важно ещё **сохранить контуры клеток и ядер**. Поэтому мы написали гибрид:

```
loss = L1(out, clean) + 0.15 · L1( Sobel(out), Sobel(clean) )
```

- **L1(out, clean)** — средняя абсолютная разница по всем пикселям. Учит сеть правильному цвету.
- **Sobel** — оператор градиента, выделяет границы. `L1` на градиентах — штраф за размытые края.
- Коэффициент **0.15** подобран опытно: при больших значениях сеть начинает «обводить» всё жирными линиями; при меньших — границы теряют чёткость.

Реализация Sobel у нас ручная (диагональные разности между соседними пикселями) — это два вычитания тензоров и `torch.abs`.

### 4.4 Цикл обучения (наш)

Сам цикл написан руками:

```python
for epoch in range(epochs):
    model.train()
    for degraded, clean in train_loader:
        optimizer.zero_grad()
        restored = model(degraded)
        loss = content_loss(restored, clean) + 0.15 * edge_loss(restored, clean)
        loss.backward()
        optimizer.step()
    # валидация
    model.eval()
    with torch.no_grad():
        for degraded, clean in val_loader:
            ...
            psnr = compute_psnr(restored, clean)
            ssim = _simple_ssim(restored, clean)
    # запись в training_status.json (наш код)
    write_status({
        "currentEpoch": epoch + 1,
        "currentLoss": loss.item(),
        "currentValLoss": val_loss,
        "psnr": psnr,
        "ssim": ssim,
    })
```

Главное — **после каждой эпохи** мы записываем прогресс в `training_status.json`. Java-сторона (UI дашборда) читает этот файл через endpoint `/training/status` и показывает прогресс-бар в реальном времени. Этой связки «пишем после каждой эпохи → Java полл-ит → рисуем бар» нет ни в одной библиотеке — это мы построили сами.

Метрики **PSNR** и **SSIM** тоже считаем сами:

- **PSNR** — простая формула `10·log10(1 / MSE)`.
- **SSIM** — упрощённая реализация `_simple_ssim` через среднее/дисперсию/ковариацию без скользящего окна. Это наш сокращённый вариант, подходящий для валидации.

### 4.5 Инференс-пайплайн (`_enhance_with_neural_model`)

Когда гистолог нажимает «Улучшить изображение», Java шлёт файл в Python, и там выполняется наш инференс-пайплайн:

1. **Декодирование** файла в PIL-картинку, нормализация `[0..255] → [0..1]`, преобразование в тензор.
2. **Паддинг до кратного 8.** Почему 8? В U-Net три уровня пулинга 2×2 — значит высота/ширина должны делиться на `2³ = 8`, иначе размеры тензоров на encoder и decoder не совпадут. Мы написали функцию `_pad_to_multiple`: отражательно дополняет изображение до ближайшего большего кратного 8.
3. **Инференс** под `torch.no_grad()` — без накопления градиентов, быстрее и меньше памяти.
4. **Un-pad** — обрезаем добавленные пиксели обратно.
5. **Денормализация** `[0..1] → [0..255]`, конвертация в `uint8`.
6. **Возврат** в исходные размеры исходной картинки (если они отличались).
7. **Выбор формата вывода**: PNG или JPEG в зависимости от MIME-типа или расширения исходного файла.

Ни один шаг этого пайплайна не является «вызовом одной функции из библиотеки» — всё склеено руками.

### 4.6 Baseline-улучшение (не нейросеть)

Параллельно с U-Net мы сделали **baseline-алгоритм** — четыре детерминированных фильтра из Pillow. Это наш «эталон»: быстрый, простой, не требующий весов.

```python
def _baseline_enhance(image):
    image = ImageOps.autocontrast(image)          # растянуть гистограмму
    image = image.filter(ImageFilter.UnsharpMask) # резкость краёв
    image = ImageEnhance.Contrast(image).enhance(1.08)   # контраст +8%
    image = ImageEnhance.Sharpness(image).enhance(1.15)  # резкость +15%
    return image
```

**Важная мысль для защиты:** здесь нет обучения, потому что **нет переменных параметров**. Коэффициенты 1.08, 1.15 и формула autocontrast — всё это константы, прошитые в библиотеке. Baseline всегда работает одинаково, предсказуем на любом компьютере, но и результат у него всегда один и тот же.

U-Net другой — у него **~11 миллионов весов**, которые изначально случайны. Сеть на старте выдаёт шум. Эти 11 миллионов чисел надо подобрать — вот этим и занимается обучение. Нет обучения — нет модели, U-Net бесполезен.

**Вот почему одной сети обучение не нужно, а другой — нужно.** Baseline — это алгоритм. U-Net — это алгоритм + таблица из 11 млн чисел, которую надо заполнить.

### 4.7 REST-слой (`app.py`)

Наш FastAPI-слой с 8 endpoint-ами:

| Метод | Endpoint | Логика |
|---|---|---|
| `GET`  | `/health` | Жив ли сервис, какое устройство (CPU/GPU), загружены ли веса |
| `GET`  | `/models` | Список моделей: всегда baseline + neural, если есть веса |
| `GET`  | `/metrics` | Метрики активной модели (loss, val_loss, PSNR, SSIM) из `latest_metadata.json` |
| `GET`  | `/training/status` | Текущий статус обучения (idle / running / ok / error), PID, текущая эпоха, loss |
| `POST` | `/training/reset-status` | Принудительно перевести статус в idle (когда зависло) |
| `GET`  | `/training/history` | История последних 20 запусков обучения |
| `DELETE` | `/clear-history` | Очистить историю |
| `POST` | `/train` | Запустить новое обучение (принимает epochs, batch_size, lr, image_size) |
| `POST` | `/enhance` | Улучшить изображение. Параметр `mode=auto/neural/baseline` |

В каждом endpoint-е мы:
- Парсим параметры FastAPI-роутами.
- Для `/enhance` принимаем `UploadFile`, читаем содержимое, определяем MIME-тип, прогоняем через выбранный пайплайн, отдаём байты с заголовком `X-Model-Name` (чтобы Java знала, какой моделью обработано).
- Обрабатываем ошибки: валидация → 400, конфликт (обучение уже идёт) → 409, внутренняя → 500.

### 4.8 Управление асинхронным обучением

Обучение длится часами. Нельзя держать HTTP-запрос открытым всё это время. Поэтому мы сделали так:

1. `POST /train` запускает **subprocess** — отдельный Python-процесс, который выполняет `train.py` с переданными параметрами.
2. На Windows subprocess запускается с флагом `CREATE_NEW_PROCESS_GROUP` — это нужно, чтобы когда uvicorn получает `CTRL_CLOSE_EVENT` при закрытии консоли, сигнал не убивал процесс обучения.
3. В `training_status.json` сразу пишется: `status=running`, `pid=...`, `startedAt=...`.
4. Обучающий процесс перезаписывает этот файл после каждой эпохи (см. 4.4).
5. При запросе `/training/status` мы не только читаем файл, но и **проверяем, жив ли PID** в операционной системе (через `os.kill(pid, 0)`). Если процесс умер, а статус всё ещё `running` — статус нормализуется в `error` или `idle`.
6. По завершению обучения subprocess дописывает итоговые метрики и помечает статус `ok` или `error`.
7. Файл `training_history.json` хранит последние 20 запусков — для списка в UI.
8. Файл `latest_metadata.json` — метрики активной модели.
9. `stdout` и `stderr` subprocess-а перенаправляются в файл `training.log`, чтобы их можно было посмотреть после падения.

Это **наша оркестрация**. FastAPI такого не умеет из коробки.

### 4.9 Java-клиент (Spring Boot)

Две сервис-обёртки:

**`AutoencoderClientService.java`** — 10 HTTP-вызовов к Python через `RestTemplate`:
- Два разных таймаута: обычный (3s connect / 5s read) для быстрых endpoint-ов и увеличенный (10s read) для `/training/status` и `/training/history`, которые могут быть медленнее.
- Для `/enhance` собирается `MultipartFile` с полем `mode` и файлом.
- Из response header читается `X-Model-Name` — чтобы потом записать в `ImageProcessingLog`, какой моделью улучшен конкретный снимок.

**`AutoencoderTrainingService.java`** — синхронизация Python-состояния с JPA-сущностью `TrainingSession`:
- При `startTraining()` создаётся запись `TrainingSession` в MySQL с `triggeredBy = currentEmployee`, статусом `RUNNING`, параметрами.
- Периодически вызывается `refreshTrainingSessionsFromPython()`: запрашиваем `/training/history`, сопоставляем с открытыми `RUNNING`-сессиями по timestamp (формат `dd.MM.yyyy HH:mm`), если находится совпадение с финальным результатом — переписываем статус на `OK`/`ERROR` и заполняем метрики.
- Сопоставление по timestamp — наше решение. Python-сторона не знает про Java-ID, а Java не хочет хранить PID Python-процесса. Общее — время запуска.

## 5. Почему baseline не обучается, а U-Net — обязательно

Повторим главную мысль отдельным блоком:

|   | Baseline | U-Net |
|---|---|---|
| Что это | Последовательность из 4 фильтров Pillow | Свёрточная нейросеть |
| Количество параметров-переменных | **0** | **≈ 11 000 000** |
| Состояние при первом запуске | Работает идентично каждый раз | Выдаёт шум (веса случайные) |
| Что нужно, чтобы заработало | Ничего, код запускается сразу | Подобрать 11 млн весов ← это и есть обучение |
| Можно ли улучшить «на месте» | Только переписать константы в коде | Дообучить на новых данных |
| Предсказуемость | Абсолютная | Зависит от того, на чём обучено |

**Обучение — это оптимизация**. Мы минимизируем нашу гибридную loss-функцию (см. 4.3) по весам сети. Оптимизатор `Adam` на каждом шаге градиентного спуска двигает все 11 млн чисел в ту сторону, где loss уменьшается. Через тысячи итераций (15 эпох × десятки тысяч пар) сеть «запоминает», как выглядит чистый гистологический кадр.

У baseline нечего двигать. Автоконтраст и unsharp mask — это математические формулы, их параметры (например, радиус размытия) зашиты Pillow раз и навсегда.
